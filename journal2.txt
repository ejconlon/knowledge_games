Chess playing
=============
I'm afraid I spent far more time programming during this period than playing actual chess (a few hours of playing), so my progress is pretty slow there.  I hope to remedy that next time, possibly taking a break from programming to read some chess books and play more games. I am satisfied that I am at a point where I can try to integrate new chess knowledge and experiment with different heuristics in my engine, which was a short-term goal of mine in this project.  Now it is incumbent upon me to gain that knowledge...


Chess move enumeration (and shortcomings)
=========================================
The first step to making my computer play chess is having it know what it can do.  At the end of the last checkpoint, I was able to determine which moves were legal (except for en passant, which I still don't have in there, but more on that soon) without any regard to check or stalemate.  It turns out that moving in chess is a very complicated operation: first, it's not "context-free" - moves like en passant require that you know the last move, not just the current state of the board.  Second, because of Castling, you are not always moving just one piece.  And third, you have to search all of your opponents next moves to make sure that you're not putting your king in check.  I put the pieces in place for determining the allowability of en passant moves (setting the last move delta and turn last moved on) but have not added the logic to yield that move in a pawn's generation of moves. I have also not yet added castling to the King's generation of moves.  (In my opinion, these things are tricky, not-fun-or-interesting cascades of if-else that I can only implement so much of at a time...)  I'm not doing any sensible ordering of moves at this point, but I will need to for alpha-beta pruning.  I'm planning on ordering capture moves first, of course, but I would need a more sophisticated heuristic to order the rest - perhaps assigning weight to center control, moving pieces off the back rank, pushing the king's moves back (since I've seen my engine move kings with little regard to their preciousness).  For now, I simply randomly shuffle the moves before I evaluate them so I get some variety in openings, and so on.


Heuristic evaluation and search in Tic-Tac-Toe 
==============================================
As you know, I have implemented the silly game Tic-Tac-Toe in order to bootstrap new interfaces and test new algorithms.  I first created a simple heuristic value class that corresponds to the extended real numbers (R U {+infinity, -infinity}), where certain victory has infinite utility, and certain loss negative infinite utility.  My heuristic for evaluating a position was the same one found in the book (if I remember correctly): 2*X_2 + X_1 - 2*O_2 - O_1 for X, and its inverse for O.  It was then easy to use the "Negamax" form of the "Minimax" algorithm.

Here's a game with only heuristic evaluation of the resulting board positions (depth 1 searching?). X is a heuristic agent, and O is a random agent:

Challenges in searching
=======================
After I got minimax and heuristic agents working in TTT, my work applying them to chess could proceed.  However, it was not entirely straightforward. For some reason I had the "brilliant" idea to throw a "MatedException" in the bowels of a recursive get_moves function when it was the case that one player was checkmated.  This would result in the game (ostensibly correctly) ending in a victory for the other player, but it wouldn't let me play out the rest of the game, since the top level move generating function would be interrupted.  The silly software engineering decision that made exception-throwing necessary was one to make move generation a composition of functions that accept and generate a list of moves.  Some functions expanded pawn promotion, some pruned moves that put the king in check, and so on.  The solution would be to make move generation a composition of functions that accept and generate tuples of (move, game state) where check and stalemate are game states that would propagate "up" the tree in some way.  I was not able to address this issue this checkpoint, so you'll see the example game below end strangely, several moves before it is really obvious that it is a checkmate (if it is indeed a checkmate - this is software in progress, you see).


Functional programming to the rescue?
=====================================
Let me say something that should have been totally obvious from the start: this engine is going to be SLOW.  There is a huge search space in the game of Chess, of course, and in the course of searching that space, you tend to generate a lot of objects and garbage.  There are two ways to maintain board state as you traverse the search tree (as I see): maintaining a mutable board that you undo changes to across every node of the tree, or maintaining an immutable board that you somehow copy across every node of the tree.  Because the former just sounds like a sure source of errors (and would be very inflexible when switching search types, board variables, etc.), I chose the latter.  My initial solution was to use Python's copy.deepcopy(object) function to create an entirely new copy of the board and all its metadata and pieces, but this was SO SLOW.  

Taking a page from my current favorite CS book, Okasaki's Purely Functional Data Structures, and with inspiration from Clojure's PersistentVector, I decided to move to persistent data structures.  This meant two things: every Board object and subobject needed to have a "copy" factory method whose job was to provide a new instance with new reference variables to old data, and every data structure needed to return a modified copy of itself on any mutating operation.  The "result" method of the Board object (which applies a move to a board) would then copy a board, perform mutating operations on its persistent data structures, and reassign the board's reference variable to the mutated structures.  This really helped to speed things up.  

If I were to start the project over again, I would give serious thought to doing it in Clojure or Haskell, since their persistent data structure support is excellent and they have strict (but useful) reference mutation semantics.  Types would also be handy.
